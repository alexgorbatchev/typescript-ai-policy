# AGENTS

This repository publishes shared **Oxfmt** and **Oxlint** configuration plus a local **Oxlint JS plugin** under `src/oxlint/`.

Subdirectory instructions take precedence inside their own scope:

- `src/oxlint/AGENTS.md` for plugin-entry and config wiring guidance
- `src/oxlint/rules/AGENTS.md` for individual rule-module and rule-test guidance

## Project intent

This project does **not** treat lint rules as generic style preferences.

The main purpose of the custom `@alexgorbatchev/*` rules is to encode **codified policies for LLM-generated and LLM-modified code** so that agent output is:

- deterministic
- reviewable
- structurally consistent
- easier to repair automatically
- less likely to hide architectural drift behind superficially valid TypeScript

Upstream Oxlint / TypeScript / Jest rules remain enabled as baseline correctness guardrails, but the custom policy layer exists primarily to steer **agent behavior**, not to enforce aesthetic preferences.

## Purpose of codified policies in this project

When you add, change, or evaluate a policy in this repository, judge it against this standard:

> A codified policy belongs here only if it meaningfully improves the reliability of code generated by LLM agents or the reliability of reviewing and repairing that code.

That means:

- prefer rules that eliminate common agent failure modes
- prefer rules that force explicit ownership, naming, and file-role decisions
- prefer rules whose violations can be explained as a concrete next action
- reject rules that are mainly stylistic, subjective, or human-taste-driven
- reject rules that are easy for an agent to satisfy mechanically without improving the underlying code

A good policy in this repo is one that turns a vague convention into a machine-checkable contract that a coding agent can follow repeatedly.

## Policy design principles

When proposing a new codified policy, document the specific agent failure mode it addresses.

Examples of acceptable motivations:

- agents repeatedly create ad-hoc local contracts instead of reusing owned types
- agents leak test helpers into runtime modules
- agents place files in non-canonical locations that make later automation unreliable
- agents satisfy type-checking while weakening semantic clarity or ownership boundaries
- agents create structurally valid but repository-invalid React, test, fixture, or import layouts

Examples of unacceptable motivations:

- the team likes one style better than another
- the code looks cleaner to a human reviewer but there is no agent-reliability gain
- the rule bans a construct without improving correctness, maintainability, ownership, or repairability

## LLM-oriented TypeScript policy stance

For this repository, TypeScript-facing policies are allowed to be stricter than general human-oriented TypeScript style guides when that strictness improves agent output quality.

In particular, policies may intentionally prohibit patterns that are technically valid TypeScript if those patterns are frequently abused by LLMs to produce low-quality code.

### Explicit-type policy goal

When designing rules around type declarations, the goal is **not** nominal typing for its own sake.
The goal is to stop agents from scattering anonymous structural contracts across the codebase and to force them toward:

1. inference when the type is already known from context
2. named declarations when an explicit type is required
3. domain ownership for shared contracts
4. semantic reuse instead of copy-pasted shape reuse

### Preferred enforcement shape for agent-generated code

If this repository adds stricter type-declaration policies, they should follow this order:

1. **Prefer inference first**
   - If a type can be derived from context, the agent should not restate it locally.
2. **Ban use-site inline structural type annotations**
   - Do not allow inline object / tuple / function / union / intersection type literals at the usage site when a named declaration or inference should be used instead.
3. **Require named declarations for explicit contracts**
   - If an explicit type is needed, it must reference a named declaration that encodes intent.
4. **Require owned placement for shared types**
   - Shared contracts must live with the owning domain/module, not in a junk-drawer `types.ts` with unrelated concepts.
5. **Prefer semantic repair over mechanical extraction**
   - A violation is not truly fixed if the agent invents a meaningless alias and leaves the ownership problem unresolved.

### Why these stricter policies exist

LLM agents disproportionately generate the following failure modes:

- repeated inline shapes such as `{ id: string }` across many files
- use-site annotations where inference should have been trusted
- meaningless one-off aliases created only to silence a linter
- tuples and structural literals that hide domain purpose
- shared contracts extracted into the wrong module or a global dumping ground

Codified policies in this repo should be written to prevent those failure modes directly.

## Message-writing requirement

Lint messages in this project should read like repair instructions for an LLM agent.

A good message tells the agent:

- what is wrong
- why it violates the repository contract
- what shape is required instead
- where the code should move when ownership is part of the policy

Do not write vague messages that merely describe the AST pattern.

## Change requirements

When changing the policy surface:

- update the relevant README / AGENTS documentation in the same change
- keep examples aligned with the enforced contract
- treat rule removal or relaxation as a policy change, not as a small local tweak
- call out interactions with other rules when a contract is coupled across multiple checks

## Whole-policy compatibility requirement

When making changes to rules, config, docs, examples, or when reviewing code against this repository's policy set:

- evaluate the entire enabled policy surface as one coupled system, not as isolated rules
- identify any direct or indirect incompatibilities, impossible states, overlapping file-role requirements, or rule interactions that cannot be satisfied at the same time
- immediately report any such incompatibility to the user; do not silently pick one rule over another or proceed with partial compliance
- if a user request would create, preserve, or depend on a policy conflict, stop and explicitly tell the user that the request is incompatible with the repository contract
- offer concrete resolution paths so the policy set returns to a compatible state; for example, narrow a rule's scope, change ownership boundaries, adjust file-role contracts, or change the conflicting request
- do not implement, recommend, or leave behind a state where different parts of the policy require mutually incompatible outcomes

This requirement applies both when evolving the policy set itself and when reviewing repository code for compliance. The goal is a policy system whose rules can all be satisfied together without hidden contradictions.

## Practical review heuristic

Before adding a rule, ask:

1. What recurring LLM failure mode does this stop?
2. Can the required replacement behavior be expressed deterministically?
3. Will the rule improve reviewability and repairability of generated diffs?
4. Can an agent satisfy the rule mechanically without actually improving the code?

If the answer to (4) is yes, the policy is incomplete and needs an ownership, naming, or placement constraint in addition to the syntax check.

## Development scripts

Available scripts:

```bash
bun run check
./scripts/lint-target.sh /Users/alex/development/projects/date-maker
bun run lint:target -- /Users/alex/development/projects/date-maker
```

`check` runs:

- `oxfmt --check .`
- `oxlint .`
- `tsgo --noEmit`
- `bun test`

`lint:target` runs this repository's current Oxlint policy against another directory by:

- using `src/oxlint/oxlint.config.ts`
- disabling nested target configs via `--disable-nested-config` so results are not mixed with the target project's own Oxlint config
- changing into the target directory first, which avoids an Oxlint crash when linting a path outside the current working root
- forwarding any extra Oxlint CLI flags after the target path

## Release workflow

Releases are tag-driven and the committed package version is the source of truth.

- bump `package.json` to the target version in git first
- create and push a matching annotated tag such as `v1.0.6`
- the release workflow validates that `package.json` matches the tag version
- after validation, the workflow runs `bun run check`, publishes to npm, and creates the matching GitHub Release with generated notes

Do not rely on CI to rewrite `package.json` during release; a tag and the committed package version must already agree.
